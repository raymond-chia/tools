你現在進入「重構模式」。

重構目標：`$ARGUMENTS`

## 核心原則

1. **禁止向後相容**——不保留舊 API、不加 deprecated 標記、不做轉接層
2. **一步到位**——直接改成最終設計，不做漸進式遷移
3. **全面修改**——所有呼叫端、測試、索引文件一併更新，不留殘留
4. **先討論後動手**——方案未經用戶確認前，禁止修改任何檔案

## 流程

### 階段一：分析現狀

閱讀目標檔案及其相關檔案（呼叫端、測試、型別定義），列出：

- 目前的設計問題（每個問題一句話）
- 誰在用這些 API（列出檔案和函數名）

### 階段二：提出重構方案

針對每個問題，必須列出**至少兩個**可行方案，並附比較表（成本、靈活性、限制等面向）。禁止只提單一方案就問確認。

每個方案必須包含：

- 要改什麼（型別、函數簽名、模組結構）
- 為什麼這樣改（解決什麼問題）
- 影響範圍（哪些檔案需要跟著改）

最後給出推薦方案及推薦理由，等待用戶確認。用戶可能會：

- 全部同意
- 部分同意、部分修改
- 要求補充說明

### 階段三：執行重構

用戶確認後，按以下順序執行：

1. 修改核心型別和函數（src/）
2. 修改所有呼叫端
3. 修改所有測試
4. 執行 `cargo check -p editor` 確認編譯通過
5. 執行 `cargo test -p board` 確認測試通過
6. 更新索引文件（`.claude/rules/*-index.md`）

每個步驟完成後繼續下一步，不需要中途等待確認。
如果編譯或測試失敗，修復後再繼續。

### 階段四：收尾

完成後報告：

- 修改了哪些檔案
- 刪除了哪些已無用的型別/函數
- 新增了哪些型別/函數

## 禁止事項

- 禁止保留舊函數並標記 deprecated
- 禁止加 wrapper 函數來維持舊 API
- 禁止保留未使用的型別或 import
- 禁止跳過測試修改（測試必須反映新設計）
- 禁止在重構中偷加新功能

## 開始

現在根據目標 `$ARGUMENTS`，開始階段一：閱讀並分析現狀。

如果目標為空，先請用戶說明想要重構的檔案或模組。
