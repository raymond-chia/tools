# 遊戲設計機制文檔

## 核心願景

### 你希望遊戲的核心樂趣是什麼？

- 規劃戰術, 逐一嘗試戰術是否有效。最終發現戰術能穩定成功時會有成就感
- 需要有某種程度的隨機性, 以免像是在下棋, 並產生最優解
- 不能隨機出現敵人, 以免難以設計戰術
- 需要根據喜好戰術培養角色, 讓不同角色使用不同技能合作達成戰術

- 比方來說 pathfinder wrath of the righteous 的 through the ashes DLC 中, 會需要利用地形甚至機關來對付敵人
- 需要針對性培養角色, 才好應付無法依靠地利對付的敵人

### 你希望玩家在戰鬥中主要做什麼決策？

- 主要是根據戰術施放技能, 一步一步完成戰術
- 戰術需要考量資源分配, 比如法術會消耗 MP
- 戰術可能包含位置, 比如把敵人推下懸崖, 或者堵住隘口讓後排盡情輸出
- 重要元素: 位置（光、掩體、高地命中、夾擊、line of sight）、位移（製造地形、利用地形、元素反應）、反應動作、壓力機制、fog of war

## 回合定義

- 單位行動 到 下次單位行動前
- 單位回合開始時檢查 buff / debuff duration != 0
- 單位回合結束才減少 duration, 處理 damage over time

## 遊戲定位

**類型**：戰術規劃 + 試錯 + 協同戰鬥 + 地形利用的回合制戰棋 RPG

**參考遊戲**：

- Pathfinder: Wrath of the Righteous (地形與機關戰術)
- Divinity: Original Sin 2 (元素互動與地形利用)
- Into the Breach (完美信息與位置戰術)
- XCOM 2 (資源管理與風險評估)
- 火焰之紋章 (地形系統與角色養成)

**核心循環**：

1. 觀察戰場（敵人位置、地形、可用資源）
2. 規劃戰術（技能組合、位置利用、資源分配）
3. 執行戰術（移動、施法、團隊合作、利用地形）
4. 評估結果（成功 → 下一關 / 失敗 → 調整戰術重試）

**核心理念**：

> **針對性的策略應該能克服隨機性。**
>
> 隨機性只是讓高風險高報酬的戰術不易實現，但不會完全不可能實現。
> 玩家可以針對性地通過裝備、技能、位置等方式提高成功率，將「賭博式戰術」變成「高成功率戰術」。

## 當前設計問題診斷

### 缺失的關鍵要素

#### 1. 地形互動系統不足

**問題**：

- 雖然有 `Shove` 效果（推擊），但沒有**深淵**等致命地形
- 沒有**阻擋視線**的地形（掩體、牆壁）
- 沒有**可互動物件**（機關、陷阱、可破壞物）

**影響**：

- 無法實現「推下懸崖」、「堵住隘口」等戰術
- 缺少地形帶來的戰術深度

#### 2. 技能協同機制不明顯

**問題**：

- 沒有「狀態 + 技能」的聯動（例如：油地 + 火球 = 範圍燃燒）
- 沒有「角色 + 角色」的配合（例如：隊友包夾 → 夾擊傷害加成）

**影響**：

- 無法實現「角色使用不同技能合作達成戰術」
- 技能只是單獨使用，缺乏協同樂趣

#### 2. 資源管理層次不足

**問題**：

- MP 目前只是每場戰鬥的資源
- 沒有跨戰鬥的資源管理（MP 應該是一天的限制）
- 沒有消耗品系統（藥水、卷軸）

**影響**：

- 資源管理決策過於簡單
- 缺少「謹慎使用大招」的緊張感
- 無法實現「連續多場戰鬥」的關卡設計

#### 3. 缺少風險回報的選擇

**問題**：

- 技能設計可能缺少「穩健」vs「賭博」的選擇
- 缺少讓玩家「克服隨機性」的機制（buff、裝備等）

**影響**：

- 無法實現「針對性策略克服隨機性」的理念
- 戰術規劃深度不足

### 機制 1：地形系統（已有基礎，需擴充）

#### 目前的設計（很好，保持）

```rust
pub struct Tile {
    pub terrain: Terrain,      // 基礎地形
    pub object: Option<Object>, // 地形物件
}

// Terrain：影響移動成本
pub enum Terrain {

    ShallowWater, // 淺水，移動成本 17

    Wall,         // 牆壁，不可通行
    Tree,         // 樹木，不可通行
```

**設計優點**：

- ✅ Terrain 和 Object 分離清晰
- ✅ 可以組合（例如：山地上的牆壁）
- ✅ 邏輯簡單，容易擴充
- ✅ 不需要複雜的高度系統

#### 地形戰術範例

**戰術 1：推下懸崖**

```
佈局：
  敵人 站在 Cliff（懸崖邊）
  Cliff 旁邊是 Pit（深淵）

執行：
  1. 戰士移動到敵人旁邊
  2. 使用「Shove」技能推擊
  3. 敵人被推入 Pit
  4. 掉落即死

結果：
  - 即使是高血量 Boss，也能一擊必殺
  - 風險：需要敵人站在懸崖邊
```

**戰術 2：堵住隘口**

```
佈局：
  山谷地形，只有一條通道（兩側是 Wall）

執行：
  1. 坦克站在通道口
  2. 敵人無法繞過（Wall 不可通行）
  3. 後排安全輸出

結果：
  - 以少敵多的關鍵戰術
  - 地形成為戰術的一部分
```

**戰術 3：利用困難地形**

```
佈局：
  我方在 Plain（移動成本 10）
  敵人在 Mountain（移動成本 20）

執行：
  1. 射手在 Plain 上移動並攻擊
  2. 敵人在 Mountain 上追不上（移動慢）
  3. 風箏戰術

結果：
  - 移動力優勢轉化為戰術優勢
```

### 機制 2：命中與隨機性系統（已有，需完善）

#### 目前的命中系統（很好，保持）

你目前的百分比系統（不是骰子！）：

```rust
// 1. 擲 1-100 的隨機數
let hit_random = rng.random_range(1..=100);

// 2. 計算命中分數
let hit_score = accuracy + hit_random;

// 3. 判定
if hit_random <= 5 {
    完全閃避（大失敗，5%）
} else if hit_random > 95 {
    完全命中（大成功，5%）
} else {
    // 計算閃避
    let evade_score = hit_score - evasion;
    if evade_score <= 0 {
        閃避
    } else if hit_score - evasion - block <= 0 {
        格擋（減傷）
    } else {
        命中
    }
}
```

**這個系統的優點**：

- ✅ 直觀：不需要解釋「d20」「d6」等術語
- ✅ 可預測：玩家可以看到 accuracy、evasion、block 數值
- ✅ 有層次：大成功/大失敗、閃避、格擋、命中、爆擊
- ✅ 符合「針對性策略克服隨機性」的理念

#### 爆擊系統（已實作）

**設計原則**：

- 每個技能有 `crit_rate` 欄位（爆擊率，0-100 的百分比）
- 爆擊判定使用與命中相同的 `hit_random` 值（不另外擲骰）
- 爆擊條件：`hit_random > (100 - crit_rate)`
- 爆擊倍率：固定 2.0x（參考 PF2e/DnD 5e 設計）
- 爆擊可被格擋（經過正常防禦流程）

**範例**：

```rust
// 高爆擊率技能
let skill = Skill {
    crit_rate: Some(20),  // 20% 爆擊率
    effects: vec![Effect::Hp { value: -10, ... }],
    ...
};

// 施放技能時
let hit_random = random_range(1..=100);  // 假設擲出 85
if hit_random > (100 - 20) {  // 85 > 80，觸發爆擊
    // 傷害 -10 × 2.0 = -20
}
```

**為什麼使用固定 2.0x 倍率？**

1. 簡單直觀：玩家看到「爆擊」就知道是雙倍傷害
2. 平衡性好：參考 PF2e/DnD 5e 的成熟設計
3. 可預測性：不需要額外的倍率計算

#### 格擋系統：Block vs BlockReduction（已實作）

**設計問題**：

之前錯誤地將 `block`（格擋機率）和格擋減傷百分比混用。現已分離為兩個獨立系統：

**Block（格擋機率）**：

- 用途：影響是否觸發格擋
- 計算：`hit_score - evasion - block`
- 如果結果 ≤ 0，觸發格擋
- 來源：被動技能的 `Effect::Block`
- 性質：防禦數值，與 evasion 疊加

**BlockReduction（格擋減傷百分比）**：

- 用途：格擋成功時的傷害減免百分比
- 基礎值：10%（無被動技能時）
- 最高值：80%（上限）
- 來源：被動技能的 `Effect::BlockReduction`
- 計算：`減傷 = 傷害 × (block_reduction / 100)`

**為什麼要分離？**

1. **概念清晰**：格擋機率 ≠ 格擋減傷
2. **策略深度**：玩家可以選擇：
   - 高 Block + 低 BlockReduction：容易觸發格擋，但減傷少
   - 低 Block + 高 BlockReduction：不易觸發，但觸發時減傷多
   - 平衡投資：兩者都有一定水準
3. **養成多樣性**：不同角色定位有不同的投資策略

**範例**：

```rust
// 單位 A：高格擋機率，低減傷
Block = 100 (來自被動技能)
BlockReduction = 10 (基礎值，無投資)
→ 容易觸發格擋，但只減傷 10%

// 單位 B：低格擋機率，高減傷
Block = 20 (少量投資)
BlockReduction = 60 (大量投資)
→ 不易觸發格擋，但觸發時減傷 60%

// 單位 C：平衡型
Block = 50
BlockReduction = 40
→ 中等觸發率，中等減傷
```

**閃避與格擋疊加的設計理念**：

閃避和格擋採用**疊加判定**（而非分離判定）：

```
閃避區間: hit_score ≤ evasion          → 完全避免傷害
格擋區間: evasion < hit_score ≤ evasion + block → 減少傷害（第二道防線）
命中區間: hit_score > evasion + block     → 完整傷害
```

**為什麼採用疊加設計？**

1. **鼓勵多元投資**：如果分離判定，玩家會只投資閃避或格擋其中一種（選擇報酬率更高的）。疊加設計讓兩者都有價值：

   - 閃避：第一道防線，完全避免傷害
   - 格擋：第二道防線，當閃避失敗時減傷

2. **創造攻防對抗**：高防禦可被高 accuracy 破解，產生策略深度：

   - 防禦方：提升 evasion + block 創造難以命中的區間
   - 攻擊方：提升 accuracy 或使用高命中技能破防

3. **平衡性**：疊加不會導致無敵，因為攻擊者可以：
   - 提升自身 accuracy（永久提升）
   - 使用高 accuracy 技能（針對性破防）
   - 搭配 buff 技能提高命中

#### 傷害系統：固定傷害（設計決策）

**設計決策：採用固定傷害，不使用範圍傷害**

```rust
Effect::Hp { value: -10, ... }  // 固定 10 傷害
```

**理由**：

1. **符合核心理念「策略克服隨機性」**

   - 固定傷害讓玩家可以精確規劃戰術
   - 可以計算「需要幾次攻擊擊殺敵人」
   - 支援戰術思考，而非碰運氣

2. **已有足夠的隨機性**

   - 命中判定（accuracy + random(1-100)）
   - 閃避判定（evasion）
   - 格擋判定（block）→ 減傷 10-80%
   - 爆擊判定（crit_rate）→ 2.0x 傷害
   - 先攻順序（initiative + 1d6）

   → **同一個技能的實際效果已經有多種可能**：

   - 完全閃避：0 傷害
   - 格擋 50%：5 傷害（假設基礎 10）
   - 正常命中：10 傷害
   - 爆擊：20 傷害
   - 爆擊被格擋 50%：10 傷害

3. **電腦遊戲不需要骰子**

   - DnD/Pathfinder 使用骰子是桌遊的物理限制
   - 電腦遊戲可以把隨機性集中在「是否命中」而非「傷害多少」
   - 參考成功案例：Into the Breach（固定傷害，高評價）

4. **避免過度隨機**
   - 太多層隨機性會削弱戰術規劃的意義
   - 玩家會感到「運氣太重要」而非「策略重要」
   - 固定傷害讓失敗時更容易分析原因

**結論**：

固定傷害更符合本遊戲的設計目標。隨機性來自**命中判定**和**爆擊判定**，而非傷害本身。

#### 技能設計範例：穩健 vs 賭博

**關鍵設計原則**：每個角色都要有「穩健技能」和「賭博技能」

使用**固定傷害 + 命中率**來區分穩健與賭博：

```rust
// 戰士技能組
pub const WARRIOR_SKILLS: &[&str] = &[
    "slash",          // 穩健：80% 命中，8 傷害
    "heavy_strike",   // 平衡：60% 命中，15 傷害
    "berserk_slash",  // 賭博：40% 命中，25 傷害
];

// 法師技能組
pub const MAGE_SKILLS: &[&str] = &[
    "magic_missile",  // 穩健：100% 命中（必中），4 傷害/發
    "fireball",       // 平衡：70% 命中，18 傷害 AOE
    "meteor",         // 賭博：50% 命中，35 傷害 AOE
];
```

**設計表格**：

| 技能類型 | 命中率  | 傷害（固定） | MP 消耗 | 設計意圖         |
| -------- | ------- | ------------ | ------- | ---------------- |
| 穩健技能 | 80-100% | 低           | 少/無   | 克服隨機性的選項 |
| 平衡技能 | 60-80%  | 中           | 中      | 日常使用         |
| 賭博技能 | 40-60%  | 高           | 高      | 高風險高回報     |

**隨機性來源**：

- 穩健技能：高命中率 → 穩定，但可能閃避/格擋/爆擊
- 賭博技能：低命中率 → 不穩定，但爆擊時傷害極高

#### 「針對性策略克服隨機性」範例

**情境**：Boss 戰，Boss 有高閃避（evasion = 50）

```
策略 A：直接攻擊（隨機）
    使用「重擊」(accuracy = 30)
    hit_score = 30 + random(1-100) = 31-130
    evade_score = hit_score - 50 = -19 到 80
    → 成功率約 60%（不穩定）

策略 B：針對性策略（克服隨機性）
    1. 使用「祝福術」給自己 +20 accuracy
    2. 裝備「精準項鍊」+10 accuracy
    3. 利用「高地優勢」+10 accuracy（未來擴充）

    現在 accuracy = 70
    evade_score = 70 + random(1-100) - 50 = 21-120
    → 成功率約 95%（穩定！）

策略 C：放棄攻擊，利用地形（完全克服隨機性）
    1. 把 Boss 引到懸崖邊
    2. 使用「Shove」推下去
    → 成功率 100%（如果能引到懸崖邊）
```

**關鍵**：

- ✅ 穩健戰術必定存在（策略 B、C）
- ✅ 玩家可以選擇承受多少風險
- ✅ 針對性準備能克服隨機性

### 機制 3：資源管理系統（MP 是一天的限制）

#### 設計目標

- MP 是「一天的資源」，不是「每場戰鬥的資源」
- 玩家需要規劃「哪一場戰鬥用大招」
- 強力技能消耗大量 MP，需要謹慎使用

#### 實作建議

```rust
pub struct Unit {
    // ... 現有欄位 ...
```

#### 長休息機制

```rust
// 戰鬥結束後
pub fn end_battle(board: &mut Board) {
    // MP 不恢復
    // HP 可能部分恢復，或需要使用道具
}

// 長休息（回營地、過一天）
pub fn long_rest(board: &mut Board) {
    for unit in board.units.values_mut() {
        unit.mp = unit.max_mp;  // MP 恢復到滿
        unit.hp = unit.max_hp;  // HP 恢復到滿
    }
}
```

## 規則系統建議

### ❌ 不要做的事

**1. 不要直接套用 PF2E 或 DND 規則**

原因：

- **版權風險**：WotC 對規則有版權保護
- **過於複雜**：PF2E Remaster 有上千條規則，小團隊無法實作
- **不適合電子遊戲**：這些規則為桌遊設計，很多機制在電子遊戲中體驗不佳
- **缺乏差異化**：你的遊戲會變成「劣質版 PF2E」

**2. 不要使用骰子術語（d20, d6 等）**

原因：

- 不是所有玩家都熟悉桌遊術語
- 百分比和範圍（5-10）更直觀
- 你的系統已經用百分比，不需要改成骰子

**3. 不要追求數值平衡（初期）**

原因：

- 先追求「有趣的戰術」，再調整平衡
- 不平衡的機制反而能創造獨特戰術（例如：推下懸崖是「不平衡」的，但很有趣）

**4. 不要做複雜的高度系統**

原因：

- 實作複雜，邏輯難懂
- Terrain + Object 的分離已經足夠
- 掉落傷害計算沒必要，直接「掉落即死」更有戰術意義

**1. 借鑒精神，不抄規則**

## 實作優先級

### 第一階段：地形戰術基礎（立即可做）

**目標**：讓「推下懸崖」戰術可玩

**步驟**：

1. 在 `Object` 中加入 `Pit` 變體
2. 修改 `Object::is_passable()`，Pit 不可通行
3. 在 `apply_effect_to_pos` 或新增函數中，處理 `Effect::Shove`：
   - 計算推擊方向和目標位置
   - 檢查目標位置是否為 `Pit`
   - 如果是 `Pit`，目標單位 `hp = 0`
4. 在 editor 中設計測試關卡：
   - 地圖有 Cliff（懸崖邊）和 Pit（深淵）
   - 敵人初始位置在 Cliff 附近
   - 玩家需要用 Shove 推敵人入 Pit

**預期成果**：

- 玩家可以把敵人推下懸崖
- 感受到「利用地形的成就感」
- 驗證「針對性策略克服隨機性」的理念

### 第二階段：MP 資源管理系統

**目標**：讓「資源分配決策」有意義

- 透過 buff 技能，可以將「賭博技能」變成「高成功率技能」
- 驗證「針對性策略克服隨機性」的理念

**步驟**：
